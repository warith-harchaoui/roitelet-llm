# Image Generation in Roitelet

Roitelet routes image-generation prompts the same way it routes text:
the strongest registered model wins, candidates without an API key are
auto-pruned, and the pipeline is **local-first** (a local Stable
Diffusion server is preferred over a paid one when ``independence`` is
on). The output is **K=1** by design — image fusion is not a
well-defined operation, so the strongest single candidate's output is
the answer.

This guide walks through wiring up an image generator. The architectural
walk-through lives in [docs/ARCHITECTURE.md](ARCHITECTURE.md).

---

## TL;DR

```bash
# Paid path — DALL-E / gpt-image-1
echo 'OPENAI_API_KEY=sk-...' >> .env

# Local path — AUTOMATIC1111 SD-WebUI with the OpenAI-compat extension
# (or llama.cpp's image-gen variants)
echo 'OPENAI_COMPATIBLE_BASE_URL=http://localhost:7860/v1' >> .env
echo 'OPENAI_COMPATIBLE_API_KEY=any-non-empty-string' >> .env

# Tell Roitelet the model exists + how good it is.
# Add a bootstrap entry with image_gen > 0 (see "Registering a model").
```

Then hit the endpoint:

```bash
curl -X POST http://localhost:8000/api/images \
  -H "Content-Type: application/json" \
  -d '{"prompt": "A small wren in a forest, oil painting", "size": "1024x1024"}'
```

The response carries the on-disk path of the generated image:

```json
{
  "conversation_id": "...",
  "model_id": "openai/gpt-image-1",
  "provider": "openai",
  "images": [
    {"path": "/.../data/images/abc123.png", "model_id": "openai/gpt-image-1",
     "provider": "openai", "revised_prompt": null, "error": null}
  ],
  "latency_s": 6.4,
  "cost_usd": 0.0
}
```

Files are served straight from disk via the FastAPI static mount at
`/data/images/<uuid>.png`.

---

## Endpoints

| Path | Body shape | Notes |
|---|---|---|
| `POST /api/images` | `ImageGenRequest` — `prompt`, `size`, `n`, `preferences` | Native endpoint. Honours `independence` and `max_cost_usd`. |
| `POST /v1/images/generations` | `OpenAIImagesRequest` — `prompt`, `model`, `n`, `size` | OpenAI-compatible. Clients that already speak this shape work unchanged. |

Both endpoints are gated by `ROITELET_API_TOKEN` when set, the same as
the chat endpoints.

## Registering an image-gen model

Image-gen models are recognised by their `image_gen` capability prior
being **strictly greater than zero**. Add a bootstrap entry:

```json
"openai/gpt-image-1": {
  "provider": "openai",
  "local": false,
  "vlm": false,
  "pricing": {"input_per_1k": 0.04, "output_per_1k": 0.0},
  "latency_s": 8.0,
  "energy_kwh": 0.0008,
  "capabilities": {
    "coding": 0.0, "math": 0.0, "reasoning": 0.0,
    "writing": 0.0, "analysis": 0.0,
    "vision": 0.0, "multilingual": 0.0, "long_context": 0.0,
    "image_gen": 0.94
  }
}
```

The `pricing.input_per_1k` field stands in for "USD per image-1024" —
the cost-budget regime treats it the same as any other paid candidate.

For an OpenAI-compatible local server (AUTOMATIC1111, ComfyUI's
OpenAI-shaped extension, llama.cpp's image-gen experimental endpoints,
etc.) use `provider: "openai-compatible"` and add the model under a
`custom_engines` entry in the Settings sheet (label + base_url +
api_key + models). The image-generation pipeline shares the same
multi-engine dispatch as text chat.

## Provider matrix

| Provider | Endpoint shape | Roitelet support |
|---|---|---|
| OpenAI direct (`gpt-image-1`, DALL-E 3) | OpenAI `/v1/images/generations` | First-class via `OpenAIImagesClient` |
| OpenRouter image relay | OpenAI shape | First-class |
| Together / Fireworks images | OpenAI shape | First-class (set `OPENAI_COMPATIBLE_BASE_URL`) |
| AUTOMATIC1111 + `sd-webui-openai-compatible-api` | OpenAI shape | First-class |
| Stability AI native (`/v2beta/`) | Stability custom | Not yet — needs a dedicated client. |
| ComfyUI JSON-RPC graphs | ComfyUI custom | Not yet — same reason. |

## Storage

Generated bytes land under `<ROITELET_DATA_DIR>/images/<uuid>.png` and
are served by the static mount at `/data/images/<uuid>.png`. The path
is recorded in the assistant `ConversationMessage.metadata` so
`/api/conversations/<id>` round-trips the image reference.

Image storage is not garbage-collected. If you generate a lot, a
periodic `find data/images/ -mtime +30 -delete` is the right shape.

---

## Limitations vs. the text pipeline

1. **No fusion.** K=1 always; the strongest image-gen candidate wins.
   "Aerial Triumvirate" is text-only.
2. **No Elo update.** The synthesis judge is what crowns winners on the
   text side; image generation has no judge yet, so the Elo loop is
   inactive for image-gen candidates.
3. **No multimodal-input pipeline integration.** The text-side
   `/api/chat/multimodal` accepts images *as input*; the image-gen
   pipeline produces images *as output*. They don't compose yet —
   image-to-image is on the roadmap.
