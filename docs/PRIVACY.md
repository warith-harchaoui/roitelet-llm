# Privacy — what Roitelet stores, what leaves the box

Roitelet is **local-first, not automatically local-only.** Whether
a prompt leaves your machine depends on which candidate models you
register and which mode you run in. This document spells out the
distinction in concrete terms so you can choose the configuration
that matches your threat model.

> ⚠️ If you only remember one sentence: **a prompt sent to a remote
> candidate goes to that candidate's provider, exactly the same way it
> would if you called that provider directly.** Roitelet does not
> obfuscate, anonymise, or sanitise the content of candidate calls.

---

## 1. The three modes

| Mode | What stays local | What goes out | When to use |
|---|---|---|---|
| **Local-only** | Everything: prompt, candidates, judge, telemetry, embeddings. | Nothing. | Sensitive prompts; air-gapped or offline use. |
| **Local-first** (default) | Synthesis judge + storage + embeddings. | The prompt is sent to each selected remote candidate (if any) using that provider's API. | Most desktop / single-user use. The trade is that remote candidates can substantially improve answer quality. |
| **Remote-augmented** | Judge + storage stay local. | Most candidates are remote; the prompt is sent to several providers per turn. | When you actively want diverse remote opinions and accept the corresponding API spend and privacy cost. |

### How to switch to local-only

Three equivalent ways:

- **Per turn:** prefix the prompt with `/local`. The slash command
  forces `independence=True` for that turn, dropping every non-local
  candidate before scoring.
- **Persistent UI toggle:** open the web Settings sheet, enable
  *Local models only*. Saved on disk; applies to every turn until you
  flip it back.
- **Auto-pruning of providers without keys.** If you don't set any
  remote API keys at all (`OPENROUTER_API_KEY`, `OPENAI_API_KEY`,
  `OPENAI_COMPATIBLE_API_KEY`, …), the registry drops remote model
  specs whose provider has no key
  (`core/registry.py::_prune_unauthorized_remotes`). Effectively
  local-only by configuration absence.

When local-only mode is active, you should see only `ollama/...`
model ids in the router decision's `selected_model_ids`. That's the
honest check.

## 2. What goes over the network when a remote candidate is selected

For each selected remote candidate on a turn, Roitelet issues one
HTTP POST to the provider's `/v1/chat/completions` (or equivalent)
endpoint, containing:

- The full **user prompt** as the `messages` field.
- The model id and any standard OpenAI-shape sampling parameters.
- The **API key** in an `Authorization: Bearer ...` header.

No additional sanitisation or redaction is applied. If the prompt
contains an attachment-derived text block (audio transcript, PDF
text, image caption), that text is in the prompt the same way it
would be if you sent it yourself.

The multimodal **inputs** themselves (audio bytes, image bytes, PDF
bytes) **never leave the machine**: they are converted to text
locally by `core/multimodal/{audio,image,pdf}.py` before the
text-only pipeline runs. The same applies to the image
*generation* pipeline (a separate endpoint): generated bytes stay
local unless the remote image-gen provider returns them as a URL,
which we then re-download (also local-bound).

### What does not go to remote providers

- The candidate replies of *other* models in the same turn. Each
  remote candidate sees only the user prompt.
- The judge's reasoning. The judge runs locally on Ollama by default;
  its system prompt + the candidate answers stay on the box.
- The rolling Elo state.
- Stored conversations.

## 3. What Roitelet stores on disk

Persistent state lives under `<ROITELET_DATA_DIR>/`:

| Path | Contents | Notes |
|---|---|---|
| `data/bootstrap/model_priors.json` | Curated capability priors per model. | Read-only at runtime. Shipped with the repo. |
| `data/conversations/<uuid>.json` | One file per chat thread: user + assistant messages, full content. | UUID-keyed; the storage layer rejects traversal payloads. |
| `data/telemetry/<uuid>.json` | One file per turn. **Includes the raw prompt, every candidate's raw reply, and the fused synthesis content.** | This is the audit trail the rolling Elo loop and the ablation runner depend on. |
| `data/runtime/elo_state.json` | Rolling Elo numbers per `(model, capability)`. | Mutated by every turn. |
| `data/runtime/settings.json` | UI-edited app settings, **including API keys**. | The HTTP `GET /api/settings` masks keys (`••••••••`) before returning them; the on-disk file holds the real value. |
| `data/cache/<provider>.jsonl` | Provider response cache, gated by `ROITELET_PROVIDER_CACHE_TTL` (default 0 = off). | Off by default. When enabled, it persists raw provider responses to disk. |
| `data/images/<uuid>.png` | Generated images (image-gen pipeline). | Served at `/data/images/<uuid>.png` by the FastAPI static mount. |
| `data/personal/inbox/` and `data/personal/wiki/` | Your personal-mode files. | Inbox is what you drop; wiki holds the converted + hand-written `.md` files. |

> The single most important line: **`data/telemetry/*.json` is full
> prompt + full responses, including from remote providers.** If you
> need to share telemetry with someone, redact it first or share only
> derived metrics.

### What is **not** stored

- Provider API keys are stored locally in `settings.json` and never
  sent to anyone except the provider those keys belong to. They are
  not logged to telemetry.
- Roitelet does not phone home, ship analytics, or call any first-party
  Roitelet server. There is no such server. The only network calls are
  to the providers and models you explicitly configured.

## 4. Practical recommendations

- **Don't commit `.env`** or `data/runtime/settings.json`. The shipped
  `.gitignore` excludes both, but verify before pushing a fork.
- **Set `ROITELET_API_TOKEN`** on any LAN deployment so the listing
  endpoints aren't open to anything on the network that can reach the
  host. The token gates `/api/settings`, `/api/conversations`,
  `/api/telemetry`, `/api/chat`, `/api/personal*`, `/api/images`, and
  `/v1/chat/completions`. `/healthz` and the static SPA stay public.
- **Periodically rotate** the `data/telemetry/` and
  `data/conversations/` directories if you want a clean slate; the
  pipeline doesn't garbage-collect them.
- **Inspect a turn before trusting fusion.** `data/telemetry/*.json`
  shows the router decision and every candidate response verbatim.
  Bad fused answers usually have a visible cause there.
- **For genuinely sensitive prompts**, run in local-only mode (`/local`
  or the settings toggle) and ideally pull the OSS bundle so the
  pipeline isn't tempted to route to anything else.

---

## 5. What "private by design" would mean (and why we don't claim it)

A system can credibly call itself private by design if:

- All data paths are local-only by construction, with no opt-out.
- All persisted state is encrypted at rest.
- No third party ever sees prompt content.

Roitelet is none of those by default. It is **local-capable**: every
piece of the pipeline (router, synthesis, embeddings, storage) can
run on your machine, and you can configure it that way. But the
default ships with optional remote candidates because that's the
configuration most users want when they install it.

If your threat model is "I don't trust LLM providers with any of my
prompts", you can reach that posture in Roitelet — by switching to
local-only mode and unregistering remote providers — but it requires
a deliberate choice, not just trust in a label.
