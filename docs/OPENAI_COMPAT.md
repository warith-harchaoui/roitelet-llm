# OpenAI-compatible API â€” drop Roitelet into your existing tooling

Roitelet ships an OpenAI-shape API at `/v1/...` so any client that
already speaks the OpenAI Chat Completions or Images Generations
protocol can use Roitelet without rewriting integration code. That
covers the official `openai` SDK, LiteLLM, Continue.dev, Cline,
LangChain's `ChatOpenAI`, and most "BYO endpoint" wrappers.

This guide is the cheat sheet for those cases.

---

## Endpoints

| Endpoint | What it does |
|---|---|
| `GET  /v1/models` | List the meta-model id (`roitelet`) plus every concrete model the registry knows about. |
| `POST /v1/chat/completions` | Run a full Roitelet turn â€” router fan-out + judge + (optional) pseudonymization. Returns the standard OpenAI shape; Roitelet-specific data lives on `roitelet_metadata`. |
| `POST /v1/images/generations` | Run a Roitelet image-gen turn (K=1, no fusion). Returns the standard OpenAI shape with paths to generated bytes. |

Streaming (`stream: true`) is supported on `/v1/chat/completions`.

---

## Minimal Python example

```python
from openai import OpenAI

client = OpenAI(
    base_url='http://localhost:8000/v1',
    api_key='sk-anything',   # ignored unless ROITELET_API_TOKEN is set on the server
)

response = client.chat.completions.create(
    model='roitelet',
    messages=[
        {'role': 'system', 'content': 'You are a concise assistant.'},
        {'role': 'user',   'content': 'Explain quicksort in one paragraph.'},
    ],
)
print(response.choices[0].message.content)
```

That's it. Roitelet flattens the message thread (system / user /
assistant / tool â€” all of them), runs its router + judge, and
returns the fused answer in OpenAI's standard envelope.

## Roitelet extras â€” `metadata.roitelet`

OpenAI's API allows arbitrary `metadata` on every request. Roitelet
reads its own knobs from a `roitelet` sub-object so clients can opt
into per-turn behaviour without dropping into the native API:

```python
response = client.chat.completions.create(
    model='roitelet',
    messages=[{'role': 'user', 'content': 'Refactor this module.'}],
    metadata={
        'roitelet': {
            'independence': True,        # only models on your machine
            'pseudonymize': True,        # hide PII before fan-out
            'top_k': 3,                  # fan out to 3 candidates
            'max_cost_usd': 0.005,       # per-turn budget filter
            'ecofrugality': 0.4,         # weight low cost + energy
            'raw_power': 0.6,            # weight answer quality
        },
    },
)
```

Unknown keys are silently ignored, so future Roitelet versions can
add fields without breaking older clients. Missing keys keep the
server-side defaults.

## What `roitelet_metadata` carries on the response

Every `/v1/chat/completions` response includes a `roitelet_metadata`
field that mirrors the native `ChatResponse`:

```jsonc
{
  "id": "chatcmpl-...",
  "object": "chat.completion",
  "model": "roitelet",
  "choices": [{"index": 0, "message": {...}, "finish_reason": "stop"}],
  "usage": {...},
  "roitelet_metadata": {
    "conversation_id": "...",
    "router": {...},          // picked models, scores, reasoning
    "responses": [...],       // per-candidate output + latency + cost
    "synthesis": {...},       // judge transcript + winners
    "telemetry_id": "...",
    "total_latency_s": 3.2,
    "pseudonymization": {...} // present only when the toggle was on
  }
}
```

Clients that don't know about this field will simply ignore it. The
existence of the field is an opt-in: anything you want to do with
the routing decision, the candidate breakdown, or the
pseudonymization audit is one extra dictionary lookup away.

## Streaming

`/v1/chat/completions` honours `stream: true` and emits standard SSE
chunks:

```python
stream = client.chat.completions.create(
    model='roitelet',
    messages=[{'role': 'user', 'content': 'Tell me a short joke.'}],
    stream=True,
)
for event in stream:
    delta = event.choices[0].delta.content
    if delta:
        print(delta, end='', flush=True)
```

Caveat: the fused answer is fully synthesised before streaming
begins. Each chunk is a slice of the final string, so wall-clock
parity with a real streaming model isn't there â€” the streaming exists
so existing clients work, not to fake progressive generation.

## Authentication

By default Roitelet does **not** require an auth header
(local-first single-user UX). When you serve Roitelet to anything
beyond `localhost`, set the env var:

```bash
ROITELET_API_TOKEN=sk-your-long-random-string
```

After restart, every `/v1/...` request must carry:

```
Authorization: Bearer sk-your-long-random-string
```

OpenAI clients send this header automatically when you set their
`api_key`. The native `/api/*` endpoints follow the same rule (the
auth dependency is shared).

## What's *not* supported

The OpenAI surface deliberately stops at the API contract; you can't
use it to:

- Fine-tune (no `/v1/fine_tuning`).
- Use the assistants API (no `/v1/assistants`, `/v1/threads`,
  `/v1/runs`).
- Use audio endpoints (Roitelet handles audio via
  `POST /api/chat/multimodal`; the OpenAI shape isn't wired up yet).
- Use embeddings (Roitelet's embedding model is reachable via the
  Ollama provider directly; there is no `/v1/embeddings` shim).
- Use function calling / tool-use as defined by OpenAI's tool schema.
  The router selects models; it doesn't pass through tool calls.

If you need any of those, the native API at `/api/*` is the right
target; the OpenAI shape is intentionally a chat-completions-only
adapter.

## Listing models

`GET /v1/models` returns the meta-id plus every concrete model in
the registry, including local Ollama models the operator has
configured:

```python
for m in client.models.list().data:
    print(m.id, m.owned_by)
```

Typical output:

```
roitelet                              roitelet
ollama/qwen3:8b                       ollama
ollama/llama3.2:3b                    ollama
openrouter/anthropic/claude-3.7-sonnet  openrouter
...
```

Clients that filter by id (e.g. "only show local models") can use
the `ollama/` prefix as the discriminator.
